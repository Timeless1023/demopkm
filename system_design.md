# 宝可梦卡交易管理系统设计（替代 Excel）

## 1. 产品定位

目标：用一个轻量 Web 系统替代 Excel，覆盖“买入-持仓-卖出-利润统计”闭环。

设计原则：
- 简单：1~2 分钟上手，字段少而清晰
- 快速：常见操作（录入一笔交易）≤ 20 秒
- 准确：利润自动计算、库存自动扣减、避免手工错账

---

## 2. 典型用户与核心流程

### 2.1 典型角色
- 店主（单人）
- 未来可扩展：员工账号（只录入）、老板账号（看利润）

### 2.2 日常流程（MVP）
1. 录入买入：卡名、版本、品相、数量、单价、来源、日期
2. 查看库存：当前可卖数量、平均成本、库存价值
3. 录入卖出：选择库存批次或自动 FIFO，输入数量、售价、平台手续费、运费
4. 系统自动计算：单笔利润、月利润、累计利润
5. 快速查询：按卡名、日期、渠道、交易类型筛选

---

## 3. MVP 功能范围（先做这个）

### 3.1 交易管理
- 买入记录（Purchase）
- 卖出记录（Sale）
- 支持修改与作废（保留审计日志）

### 3.2 库存管理
- 实时库存数量
- 库存成本（平均成本 + FIFO 成本轨迹）
- 库存预警（数量低于阈值）

### 3.3 利润核算
- 单笔毛利润 = 卖出收入 - 对应成本 - 手续费 - 运费
- 日 / 周 / 月利润报表
- 渠道维度利润（闲鱼、卡店、私聊等）

### 3.4 查询与导出
- 多条件筛选
- 一键导出 Excel/CSV（保留和 Excel 的兼容）

### 3.5 基础设置
- 卡片字典（卡名、编号、稀有度）
- 交易渠道设置
- 手续费模板（不同平台）

---

## 4. 信息架构与页面建议

## 4.1 页面结构
- 仪表盘
  - 今日销售额、今日利润、本月利润、库存总值
- 交易记录
  - 买入 / 卖出列表 + 筛选 + 新建按钮
- 库存
  - 按卡片汇总库存（数量、成本、可售价值）
- 报表
  - 趋势图、渠道对比、畅销卡排行
- 设置
  - 渠道、费用模板、用户管理

### 4.2 关键交互（强调“快”）
- 新建交易弹窗：默认值自动带出（如常用渠道、手续费率）
- 自动补全卡名：减少输入错误
- 卖出时智能提示：库存不足时禁止提交
- 批量导入：支持从历史 Excel 初次迁移

## 5. 数据模型（简化版）

### 5.1 核心实体
- `cards`
  - id, card_code, card_name, set_name, rarity
- `inventory_lots`（买入批次）
  - id, card_id, qty_in, qty_remaining, unit_cost, purchased_at, source
- `sales`
  - id, sold_at, channel, customer, shipping_fee, platform_fee, note
- `sale_items`
  - id, sale_id, card_id, qty, unit_price
- `sale_cost_allocations`（成本分摊）
  - id, sale_item_id, inventory_lot_id, qty_used, unit_cost
- `expenses`（可选）
  - id, occurred_at, type, amount, note

### 5.2 为什么要有“批次”
有批次才能准确算成本与利润（尤其同一卡不同买入价）。
建议默认 FIFO（先进先出），后续可切平均成本模式。

---

## 6. 关键计算规则（统一口径）

### 6.1 收入
- `收入 = Σ(卖出数量 × 卖出单价)`

### 6.2 销售成本（COGS）
- 按 FIFO 从 `inventory_lots.qty_remaining` 扣减
- `成本 = Σ(分摊数量 × 对应批次成本单价)`

### 6.3 单笔利润
- `单笔利润 = 收入 - 成本 - 平台手续费 - 运费`

### 6.4 月利润
- `月利润 = 月内所有单笔利润之和 - 其他运营费用（可选）`

> 强烈建议：系统内固定一套口径，避免“Excel 每个人一套算法”。

---

## 7. 技术方案建议（轻量可维护）

## 7.1 推荐技术栈（小团队友好）
- 前端：React + Ant Design（表单与表格开发快）
- 后端：Node.js (NestJS/Express) 或 Python (FastAPI)
- 数据库：PostgreSQL（关系清晰、统计查询稳定）
- 部署：Docker + 云主机（2C4G 即可）

### 7.2 为什么不直接继续 Excel
- 无法保证并发与一致性
- 难做权限与审计
- 复杂统计维护成本高且易错

### 7.3 同步与备份
- 云端部署，多设备登录
- 每日自动备份数据库
- 提供“导出 Excel”作为兜底

## 8. 实施路线图（4 周可上线 MVP）

### 第 1 周：需求确认 + 原型
- 字段冻结（买入/卖出必填项）
- 页面原型评审

### 第 2 周：后端与数据库
- 完成交易、库存、利润核心 API
- 完成 FIFO 成本分摊逻辑

### 第 3 周：前端页面
- 交易录入、库存列表、利润看板
- 筛选与导出

### 第 4 周：联调与上线
- Excel 历史数据导入
- 对账测试（抽样 30 笔）
- 上线 + 使用培训

---

## 9. 风险点与规避

- 历史 Excel 数据脏：
  - 先做清洗模板（日期、卡名、数量、金额标准化）
- 成本算法争议：
  - 上线前锁定 FIFO 口径并写入系统说明
- 用户不愿迁移：
  - 保留“导入/导出 Excel”降低切换成本

---

## 10. 你现在就可以执行的最小版本（建议）

先做 3 个页面：
1. 交易录入页（买入/卖出）
2. 库存页
3. 利润报表页

先做 5 个字段（最小闭环）：
- 卡名
- 交易类型（买入/卖出）
- 数量
- 单价
- 日期

在这个基础上再逐步加手续费、运费、渠道、客户等字段。

---

## 11. 验收标准（是否真的替代 Excel）

满足以下即可判定 MVP 成功：
- 任意一笔交易可在 20 秒内录入
- 任意卡片库存可在 3 秒内查到
- 月利润报表一键生成
- 抽样对账误差为 0（以既定口径）
- 支持导出给财务继续处理
